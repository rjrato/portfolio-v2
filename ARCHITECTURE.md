## 1. Project Overview

Personal portfolio website to showcase software engineering services and past work, and to capture client leads via a contact form.

**Target audience:** Potential clients looking for software development services (landing pages, web apps, mobile apps, AI features).

**Goals:**
- Present services clearly and convert visitors into leads
- Showcase past projects
- Provide a simple, secure contact mechanism
- Be maintainable and easy to extend by a solo developer

## 4. Routing & i18n

All routes are nested under `[lang]` to support PT-PT and EN locales.

```
/pt        в†’ Home (PT)
/en        в†’ Home (EN)
/pt/projects
/en/projects
/pt/contact
/en/contact
```

**Locale detection** is handled in `src/middleware.ts`. The middleware reads the `Accept-Language` header and redirects root `/` to the appropriate locale. `defaultLocale` is `pt`.

**Dictionaries** live in `src/lib/i18n/locales/` as TypeScript files (not JSON) for type safety. The `getDictionary(lang)` function in `src/lib/i18n/dictionaries.ts` uses dynamic `import()` and is called only in Server Components.

```typescript
// src/lib/i18n/config.ts
export const locales = ['pt', 'en'] as const
export type Locale = (typeof locales)[number]
export const defaultLocale: Locale = 'pt'
```

**Rule:** Dictionary keys must be identical across all locale files. TypeScript will catch mismatches if you export a shared `Dictionary` type from `config.ts`.

## 5. Data вЂ” Projects

Projects are defined statically in `src/lib/data/projects.ts`. No database involvement. To add or remove a project, edit this file directly.

```typescript
// src/types/project.ts
export type Project = {
  id: string          // Slug вЂ” unique, lowercase, hyphenated (e.g. "greenlink")
  title: string
  description: string
  stack: string[]
  liveUrl?: string
  repoUrl?: string
  featured?: boolean  // Controls appearance in the homepage preview
}

// src/lib/data/projects.ts
import type { Project } from '@/types/project'

export const projects: Project[] = [
  {
    id: "greenlink",
    title: "Greenlink",
    description: "...",
    stack: ["React", "TypeScript", "Next.js", "Supabase", "Vercel"],
    liveUrl: "https://greenlink.pt",
    featured: true,
  },
]
```

**Why slugs instead of numeric IDs:** Sequential numbers break when items are removed or reordered. Slugs are stable, human-readable, and URL-safe.

## 6. Database вЂ” Contacts Only

The only table in the database is `contacts`. Projects are static and do not require a database.

```typescript
// src/db/schema/contacts.ts
import { pgTable, text, timestamp, uuid } from 'drizzle-orm/pg-core'

export const contacts = pgTable('contacts', {
  id: uuid('id').defaultRandom().primaryKey(),
  name: text('name').notNull(),
  email: text('email').notNull(),
  service: text('service').notNull(),   // Selected service interest
  message: text('message').notNull(),
  createdAt: timestamp('created_at').defaultNow().notNull(),
})
```

**Migrations** are managed by Drizzle Kit (`drizzle-kit generate` + `drizzle-kit migrate`). Never modify the Supabase schema via the Dashboard вЂ” all changes go through Drizzle schema files so the Git history is the source of truth.

## 7. Contact Form Flow

The contact form is the only user-interactive feature that touches the backend.

**Fields:**
- Name (required)
- Email (required)
- Service of interest (required) вЂ” select: Landing Page & Website, Web App & MVP, Mobile App, AI Features & Automation, Other
- Message (required)

**Flow:**

```
User submits form
  в†’ Turnstile token validated server-side
  в†’ Zod validates all fields
  в†’ Insert row into Supabase contacts table (via Drizzle)
  в†’ Resend sends confirmation email to user
  в†’ Resend sends notification email to owner
  в†’ Return success state to form
```

**Server Action** (`src/actions/contact.ts`):

```typescript
'use server'
import { z } from 'zod'
import { db } from '@/db'
import { contacts } from '@/db/schema/contacts'
import { resend } from '@/lib/resend'

export const contactSchema = z.object({
  name: z.string().min(2).max(100),
  email: z.string().email(),
  service: z.enum(['landing-websites', 'web-apps-mvps', 'mobile-apps', 'ai-features', 'other']),
  message: z.string().min(10).max(2000),
  turnstileToken: z.string().min(1),
})

export type ContactFormData = z.infer<typeof contactSchema>
export type ContactState = {
  success?: boolean
  errors?: z.ZodFormattedError<ContactFormData>
  serverError?: string
}

export async function submitContact(
  prev: ContactState,
  formData: FormData
): Promise<ContactState> {
  // 1. Validate with Zod
  const result = contactSchema.safeParse(Object.fromEntries(formData))
  if (!result.success) return { errors: result.error.format() }

  // 2. Verify Turnstile token
  // ... Cloudflare verification call

  // 3. Save to Supabase via Drizzle
  await db.insert(contacts).values({ ... })

  // 4. Send emails via Resend
  // ... confirmation + notification

  return { success: true }
}
```

**Client hook** (`src/hooks/use-contact-form.ts`) uses `useActionState` to manage form state and pending state without additional libraries.

## 8. Component Classification Rules

When deciding where a component lives, apply this rule:

> **"Can this component be dropped into any page without changes?"**

| Answer | Location | Examples |
|---|---|---|
| Yes вЂ” no business logic, no page context | `components/ui/` | Button, Card, Input, SocialLinks, TurnstileWidget |
| No вЂ” knows about services, contacts, or a specific page | `components/sections/` | HeroSection, ServicesSection, ContactForm, ProjectsGrid |
| It's structural chrome | `components/layout/` | Header, Footer, ThemeToggle |

**Rule:** `page.tsx` files contain zero logic. They only compose section components and pass dictionaries.

```typescript
// src/app/[lang]/contact/page.tsx
import { ContactSection } from '@/components/sections/contact-section'
import { getDictionary } from '@/lib/i18n/dictionaries'

export default async function ContactPage({
  params,
}: {
  params: Promise<{ lang: string }>
}) {
  const { lang } = await params
  const dict = await getDictionary(lang)
  return <ContactSection dict={dict.contact} />
}
```

## 9. Theming вЂ” Dark & Light Mode

Both themes are implemented from the start using **CSS variables + Tailwind semantic tokens**. Only one theme may be visible by default (dark), but the architecture supports both with zero rework.

**Rule (non-negotiable):** Never use raw Tailwind color classes like `bg-zinc-900` or `text-white` in components. Always use semantic tokens: `bg-background`, `text-foreground`, `text-muted-foreground`, `border-border`, `bg-card`, etc.

This means adding light mode later requires only updating CSS variable values in `globals.css` вЂ” no component changes.

The `ThemeToggle` component in `components/layout/` controls the `dark` class on the `<html>` element. Theme preference is stored in-memory (no `localStorage` in sandboxed environments).

## 12. Key Conventions

- **Imports:** Always use `@/` alias, never relative paths (`../../`)
- **Component files:** kebab-case (`contact-form.tsx`), never PascalCase filenames
- **Exported components:** PascalCase (`export function ContactForm`)
- **Types:** Defined in `src/types/` or inferred with `z.infer<>` from Zod schemas вЂ” never duplicated
- **Server Actions:** All in `src/actions/`, always marked `'use server'`, always validate with Zod before any DB/email call
- **No logic in `page.tsx`:** Pages are composition only вЂ” they call `getDictionary` and render section components
- **Semantic tokens only:** Never raw Tailwind color utilities in components (see Section 9)
- **Drizzle for all DB writes:** Never use the Supabase JS client for database queries вЂ” Drizzle only. Supabase client is reserved for Auth and Storage if added in future
- **Project IDs:** String slugs, never numeric sequences
